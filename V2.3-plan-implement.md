## Critical Discovery: V1 Context Flow Already Exists

The original review stated V1 pipelines have no inter-stage context flow. **This is partially wrong.** The dispatcher at `src/orchestrator/dispatcher.ts:559-572` already assembles prior stage `resultSummary` into the next stage's input:

```typescript
// dispatcher.ts:559-572 (enqueueStepRun, else branch)
const dependsOn = parseDependsOn(stepRun.dependsOn);
const priorOutputSections = dependsOn
  .map((key) => {
    const prior = pipelineRun.stepRuns.find((s) => s.stepKey === key);
    return prior?.resultSummary ? `## Output from ${key}\n${prior.resultSummary}` : null;
  })
  .filter((s): s is string => s !== null);

enrichedInput =
  priorOutputSections.length > 0
    ? `${stepRun.inputSnapshot}\n\n---\n${priorOutputSections.join("\n\n")}`
    : stepRun.inputSnapshot;
```

**What this means for V2.3:** Step 2.1 is an improvement, not a greenfield build. Two problems with the existing code:
1. Only injects **direct** dependency outputs. Builder gets architect, but quality-guard only gets builder ‚Äî not both.
2. No truncation/cap. Verbose output bloats downstream input without limit.

**Revised Step 2.1 scope:** Add transitive ancestor collection + per-stage 4000-char / total 12000-char caps.

## Phase 2: Context Flow and Feedback Loop

### Step 2.1 ‚Äî Improve V1 Inter-Stage Context Assembly

**File:** `src/orchestrator/dispatcher.ts`, method `enqueueStepRun()`, V1 else branch (lines 559-572)

**Replace** lines 559-572 with:

```typescript
} else {
  // Collect ALL completed prior steps by walking the dependency graph backward.
  const completedPriors: { key: string; summary: string }[] = [];
  const visited = new Set<string>();
  const queue = [...parseDependsOn(stepRun.dependsOn)];

  while (queue.length > 0) {
    const depKey = queue.shift()!;
    if (visited.has(depKey)) continue;
    visited.add(depKey);

    const prior = pipelineRun.stepRuns.find((s) => s.stepKey === depKey);
    if (prior?.resultSummary) {
      completedPriors.push({ key: depKey, summary: prior.resultSummary });
    }
    if (prior) {
      for (const ancestor of parseDependsOn(prior.dependsOn)) {
        if (!visited.has(ancestor)) queue.push(ancestor);
      }
    }
  }

  // Apply per-stage cap (4000 chars) and total cap (12000 chars) per Decision 7.
  const PER_STAGE_CAP = 4000;
  const TOTAL_CAP = 12000;
  let totalLength = 0;

  const priorOutputSections: string[] = [];
  for (const { key, summary } of completedPriors) {
    const capped = summary.length > PER_STAGE_CAP
      ? `${summary.slice(0, PER_STAGE_CAP)}\n[truncated to ${PER_STAGE_CAP} chars]`
      : summary;
    if (totalLength + capped.length > TOTAL_CAP) break;
    priorOutputSections.push(`## Output from ${key}\n${capped}`);
    totalLength += capped.length;
  }

  enrichedInput =
    priorOutputSections.length > 0
      ? `${stepRun.inputSnapshot}\n\n---\n${priorOutputSections.join("\n\n")}`
      : stepRun.inputSnapshot;
}
```

**Resulting behavior for feature pipeline** (intake-gate ‚Üí architect ‚Üí builder ‚Üí quality-guard ‚Üí devops ‚Üí retrospective-analyst):
- architect receives: intake-gate output
- builder receives: architect + intake-gate
- quality-guard receives: builder + architect + intake-gate (up to 12K total)
- devops receives: quality-guard + builder + architect + intake-gate (up to 12K total)

**Edge cases:**
- Empty `resultSummary`: filtered by `if (prior?.resultSummary)`
- Circular deps: impossible in current linear DAGs; `visited` set prevents infinite loops regardless
- Single huge output: capped at 4000 chars per stage

**Add dispatcher test:** Pipeline A‚ÜíB‚ÜíC. When C is enqueued, `enrichedInput` contains output from both A and B.

**Acceptance criteria:**
- Downstream stages receive all ancestor outputs, not just direct parents
- Per-stage 4000 char cap applied
- Total 12000 char cap applied
- New test covers transitive context
- Existing tests still pass

function baseUrl(): string {
  return `http://localhost:${loadConfig().port}`;

async function api<T>(path: string, method = "GET", body?: unknown): Promise<T> {
  const res = await fetch(`${baseUrl()}${path}`, {
    method,
    headers: { "Content-Type": "application/json" },
    body: body ? JSON.stringify(body) : undefined,
  });
  if (!res.ok) {
    const err = (await res.json()) as { error: string };
    throw new Error(err.error ?? `HTTP ${res.status}`);
  }
  return res.json() as Promise<T>;
}

### Step 3.2 ‚Äî Unify MCP Fetch Helpers

**File:** `src/mcp/index.ts`

**Current state:**
- `fetchJson<T>()` (lines 12-37): throws on error, returns typed `T`
- `fetchApi()` (lines 114-154): catches errors, returns `{ isError, content }` for MCP tool responses

**These serve genuinely different purposes** ‚Äî one is typed internal, one is MCP response formatting. The duplication is in HTTP logic, not in purpose.

**Change:** Make `fetchApi` delegate to `fetchJson` internally:

```typescript
async function fetchJson<T>(path: string, options: RequestInit = {}): Promise<T> {
  const url = `http://localhost:${PORT}/v1${path}`;
  const res = await fetch(url, {
    ...options,
    headers: { "Content-Type": "application/json", ...options.headers },
  });
  if (!res.ok) {
    const text = await res.text();
    throw new Error(`${res.status}: ${text}`);
  }
  const text = await res.text();
  return text ? JSON.parse(text) : ({} as T);
}

async function fetchApi(path: string, options: RequestInit = {}) {
  try {
    const data = await fetchJson<unknown>(path, options);
    return {
      isError: false,
      content: [{ type: "text" as const, text: JSON.stringify(data, null, 2) }],
    };
  } catch (err) {
    return {
      isError: true,
      content: [{ type: "text" as const, text: String(err) }],
    };
  }
}
```

Remove the old `fetchApi` HTTP logic. Verify all MCP tool call sites still work (they return the same shape).

**Acceptance criteria:**
- One HTTP implementation, two interfaces
- All MCP tools still work
- `npm run lint` passes
