# Forge V2.3 — Project Phasing

**Source docs:** `V2.3-plan.md`, `V2.3-plan-implement.md`  
**Planning date:** 2026-03-29  
**Purpose:** Convert the implementation plan into a delivery-oriented project phasing model

---

## Planning Rule

For project execution:

- `V2.3-plan.md` is the **product and architecture intent**
- `V2.3-plan-implement.md` is the **execution detail and conflict resolver**

If the two documents disagree, the implementation guide should win for sequencing and scope. The clearest example is V1 context flow: it already exists partially, so the project work is an improvement phase, not a greenfield build.

---

## Executive Summary

V2.3 should be delivered as a **stabilization and consolidation release**, not as a feature release.

The project should be phased around four delivery outcomes:

1. Protect the critical path with tests
2. Remove dead and misleading product surface area
3. Strengthen fast-mode pipeline behavior
4. Simplify external entry points and document the final shape

This keeps the team focused on the real V2.3 promise: making Forge consistent, honest, and easier to maintain.

---

## Recommended Project Phases

| Phase | Name | Primary Goal | Depends On | Exit Gate |
|---|---|---|---|---|
| P0 | Validation & Safety Net | Lock pre-checks and add test coverage to dispatcher/worker | None | `npm run test` green with new critical-path tests |
| P1 | Surface Reduction | Remove dead execution paths, sync/auth remnants, stale UI, unused schema | P0 | `npm run lint`, `npm run build`, `npm run test` green |
| P2 | Pipeline Reliability | Improve fast-mode context propagation and enable quality feedback loop | P1 | Fast pipelines carry upstream context and can reject/revise |
| P3 | Interface Simplification | Introduce `forge run`, simplify MCP fetch flow | P2 | CLI and MCP surface reflect the new primary path |
| P4 | Documentation & Release Readiness | Rewrite human-facing architecture docs and validate release state | P3 | Docs match runtime reality and final verification passes |

## Phase Detail

### P0 — Validation & Safety Net

**Goal:** Create a safe baseline before deleting or rewiring anything.

**Scope**
- Run the pre-check list from `V2.3-plan-implement.md`
- Add dispatcher unit tests
- Add worker unit tests
- Confirm baseline build/test health

**Key outputs**
- `src/orchestrator/dispatcher.test.ts`
- `src/bridge/worker.test.ts`
- Verified answers for the 10 pre-check items

**Why this phase stands alone**
- Every later phase touches critical orchestration paths
- Without tests, Phase 1 removals are high risk

**Exit gate**
- Minimum 7 dispatcher scenarios covered
- Minimum 7 worker scenarios covered
- `npm run test` passes

**Suggested ownership**
- Track A: dispatcher tests
- Track B: worker tests
- Track C: pre-check validation and dependency confirmation

---

### P1 — Surface Reduction

**Goal:** Remove code and UI that make the product look broader than it really is.

**Scope**
- Delete bridge routes
- Delete completions endpoint
- Delete deprecated agent defaults
- Remove sync worker, login, logout
- Remove unused schema tables and relations
- Remove placeholder/stale web UI elements
- Remove `forge company` and `forge issue run`

**Sub-workstreams**
- **P1-A Runtime path cleanup**
  Remove bridge/completions and sync-related execution remnants
- **P1-B Schema cleanup**
  Remove `SyncOutbox`, `Skill`, `SkillExecution`, `AgentApiKey`
- **P1-C UI honesty cleanup**
  Remove Redis fiction from health views and remove dead budget delete action
- **P1-D CLI cleanup**
  Remove non-core commands and align to single-company local-first behavior

**Why this phase should happen before pipeline improvements**
- It reduces noise before changing orchestration behavior
- It ensures Phase 2 improves the only supported path, not one of several

**Exit gate**
- Removed files are gone
- Prisma client regenerated successfully
- `forge --help` no longer exposes deprecated commands
- Web UI builds without stale health/delete behavior
- `npm run lint`, `npm run build`, `npm run test` all pass

**Phase risk**
- Highest integration risk is schema cleanup plus sync removal
- Mitigation: keep schema edits grouped and regenerate Prisma immediately after edits

---

### P2 — Pipeline Reliability

**Goal:** Make fast-mode pipelines genuinely cumulative and enforceable.

**Scope**
- Improve V1/classic context assembly from direct-only to transitive ancestors
- Add per-stage and total caps to context injection
- Enable quality-guard loopback in feature/bugfix/refactor/release pipelines
- Update quality-guard prompt to emit structured JSON verdicts
- Extend dispatcher tests for transitive context and revision loop behavior

**Core delivery outcome**
- Fast mode stops being ceremonial
- Builder sees architect output
- Quality-guard can reject and send work back for revision

**Why this is the project’s value phase**
- This is the only phase that materially improves runtime behavior for the main user path
- Everything else mainly removes confusion or technical debt

**Exit gate**
- Downstream classic steps receive all ancestor context within caps
- Quality-guard rejection re-queues builder with revision feedback
- Max revision cap is enforced
- Tests cover both behaviors

**Phase risk**
- Token/input growth
- Verdict formatting drift from agent output

**Mitigation**
- Keep the 4000-char per-stage and 12000-char total caps
- Reuse existing `parseReviewerDecision()` contract instead of inventing a new one

---

### P3 — Interface Simplification

**Goal:** Align user-facing entry points with the consolidated architecture.

**Scope**
- Add `forge run` as the primary CLI submission command
- Keep `forge feature` and `forge bug` working as compatibility paths
- Refactor MCP fetch helper duplication so HTTP behavior has one implementation path

**Why this is separate from P1**
- These are simplifying changes, not removals
- They are safer after the runtime shape is already cleaned up and validated

**Exit gate**
- `forge run` supports `feature`, `bug`, `refactor`, `release`
- Existing familiar commands still work
- MCP tools still function after fetch helper consolidation
- `npm run lint` and `npm run test` pass

---

### P4 — Documentation & Release Readiness

**Goal:** Freeze the post-V2.3 operating model in documentation and verify release integrity.

**Scope**
- Rewrite `CLAUDE.md` around the real architecture
- Update command guidance to reflect `forge run`
- Document fast vs structured mode in their final V2.3 form
- Run final verification pass

**Why this must be last**
- Earlier documentation work will drift as removals and simplifications land
- This phase should describe the system that actually exists

**Exit gate**
- `CLAUDE.md` matches the runtime and CLI
- Final `npm run build`, `npm run test`, `npm run lint` pass
- Team can onboard from docs without relying on tribal knowledge

---

## Critical Path

The true critical path is:

`P0 -> P1 -> P2 -> P4`

P3 is important, but it is not on the architectural critical path. If schedule pressure appears, P3 can be partially deferred after P2 provided no user-facing confusion is introduced.

---

## Parallelization Model

Recommended parallel execution inside the project:

- **P0**
  Dispatcher tests and worker tests can run in parallel
- **P1**
  Bridge/completions removal, deprecated prompt cleanup, and web UI cleanup can run in parallel
  Schema cleanup and sync removal should stay coordinated
  CLI cleanup should start after sync/auth removal is stable
- **P2**
  Sequential
  Context propagation first, quality-loop second
- **P3**
  CLI `run` and MCP fetch consolidation can run in parallel
- **P4**
  Last and mostly single-threaded

## Suggested Delivery Cadence

If the team wants a lightweight release cadence, use this structure:

| Sprint / Window | Recommended Scope |
|---|---|
| Window 1 | P0 complete |
| Window 2 | P1-A + P1-B |
| Window 3 | P1-C + P1-D and verification |
| Window 4 | P2 complete |
| Window 5 | P3 + P4 + release verification |

If capacity is tighter, P1 can be split into two release branches:

- Branch 1: runtime/schema removals
- Branch 2: web/CLI cleanup

## Milestones

### Milestone M1 — Safe To Refactor
- Pre-checks completed
- Critical tests exist
- Baseline runtime behavior is observable

### Milestone M2 — Honest Product Surface
- No unsupported execution path remains
- No dead auth/sync path remains
- CLI and web UI stop advertising fiction

### Milestone M3 — Fast Mode Works As Intended
- Context truly flows across stages
- Quality review can block and force revision

### Milestone M4 — V2.3 Release Candidate
- Submission path simplified
- MCP layer cleaned up
- Documentation reflects actual system

---

## Phase Acceptance Checklist

Before declaring V2.3 complete, confirm all of the following:

- All agent execution goes through the pipeline system
- Fast mode stages consume meaningful upstream context
- Quality-guard can reject and trigger bounded revision
- Dispatcher and worker are both covered by meaningful tests
- Dead schema/models/routes/commands are removed
- Web UI contains no knowingly broken or fake elements
- `forge run` exists and works
- Docs describe the real system, not the legacy one

---

## Recommendation

Do **not** run this as one flat implementation stream of 40+ file operations. Treat it as a 5-phase stabilization project with explicit gates between phases.

That gives the team three advantages:

1. Safer rollback points
2. Cleaner ownership boundaries
3. A clearer story for stakeholders: "first make it safe, then make it smaller, then make it stronger"

This is the right delivery shape for V2.3 because the release is fundamentally about trust, not feature breadth.
