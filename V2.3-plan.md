# Forge V2.3 — Implementation Plan

**Status:** DRAFT
**Date:** 2026-03-29
**Authors:** Principal Architecture Review
**Predecessor:** V2.1 Final Architecture Decision (LOCKED, 2026-03-25)
**Purpose:** Turn the V2.3 architectural review into an executable development plan

---

## Introduction

V2.3 is a **consolidation release**. It does not add new capabilities. It removes dead code, eliminates parallel execution paths, wires real context flow into the default pipeline mode, strengthens the primary runtime surface, and adds test coverage to the most critical subsystems.

The goal is to make Forge **honest** — every feature that exists should actually work, every pipeline stage should produce real value, and every surface the user sees should be maintained and coherent.

This document is written so that an implementing engineer (human or AI) can execute each step with minimal interpretation.

---

## Architectural Goal

### What Forge should become in V2.3

A **local-first CLI agent orchestration runtime** with:

- One primary interactive surface: the **TUI console** (`forge start`)
- One integration surface: the **MCP server** (for Claude Code)
- One monitoring surface: the **web dashboard** (read-heavy, secondary)
- One pipeline system with two modes: **fast** (linear, lightweight) and **structured** (harness, multi-sprint)
- Real inter-stage context flow in **both** modes
- Typed artifact contracts in structured mode (already done in V2.1)
- Meaningful quality enforcement in fast mode (currently missing)

### What Forge should stop trying to be

- A multi-tenant hosted platform (remove company/login/logout/sync)
- An OpenAI-compatible API provider (remove completions endpoint)
- A legacy direct-agent invocation system (remove bridge routes)
- A platform that supports four equal UI surfaces (pick one primary)

---

## Current Problems

These are the problems discovered in the architectural review, ranked by severity:

1. **V1 fast pipelines have no inter-stage context flow.** The architect's output does not flow into the builder's input. Stages are ceremonial.
2. **Three parallel execution paths exist** (pipeline, bridge routes, completions endpoint). Bridge and completions bypass guardrails, budget, and artifacts.
3. **Zero test coverage** on the dispatcher and worker — the two most critical files.
4. **Memory system is fake** for cross-run learning. `MemoryEntry` table is never read during execution. `context.md` is capped at 2000 chars with no verification.
5. **Quality-guard in V1 has no feedback loop.** It runs but cannot reject or loop back.
6. **Dead code accumulation:** deprecated agent defaults, placeholder pages, unused DB tables (Skill, SkillExecution, AgentApiKey), cloud sync infrastructure with no backend.
7. **CLI surface area is confusing:** `forge feature`, `forge bug`, `forge work` overlap. `forge setup` duplicates `forge init`.
8. **Web UI has stale remnants:** placeholder.ts, Redis in HealthStatus, budget delete TODO, no workflow submission.

---

## Guiding Principles

1. **Delete before refactor. Refactor before add.** Every change should reduce total surface area or increase the honesty of existing surface area.
2. **One path, not three.** All agent execution must flow through the pipeline system. No exceptions.
3. **Context must flow or the stage is ceremony.** If a pipeline stage does not receive prior stage output, it has no business existing as a separate stage.
4. **Test what matters most.** The dispatcher and worker are the heart of the system. They must have tests before any new feature work.
5. **Honest surfaces only.** If a UI element says "delete" but the handler is a TODO, remove the button. If a table exists in the schema but no code uses it, remove the table.
6. **Local-first means local-only for now.** Cloud sync, login/logout, multi-tenancy — all deferred until V3 or later. Remove or gate the code.

## Step-by-Step Development Plan

### Phase 0: Safety Net (Tests)

Write tests for the critical path **before** making any changes. This is non-negotiable.

---

#### Step 0.1: Dispatcher Tests

**Objective:** Test the pipeline dispatcher's core logic.

**Why it matters:** The dispatcher is the most critical file in the project (1088 lines). Every subsequent change in this plan touches pipeline behavior. Without tests, we cannot verify that changes don't break existing functionality.

**What to change:** Create `src/orchestrator/dispatcher.test.ts`.

**Suggested implementation approach:**
- Use vitest (already configured in the project).
- Mock the database layer (Prisma) — don't test against real SQLite for unit tests.
- Test these scenarios at minimum:
  1. Linear pipeline: steps execute in dependency order
  2. Step failure: dependent steps are marked as `skipped`
  3. Transitive failure: if step A fails, steps B (depends on A) and C (depends on B) all skip
  4. Feedback loop: when a step with `loopsBackTo` is rejected, the target step re-queues
  5. `maxRevisions` enforcement: after N rejections, the loop stops
  6. Harness sprint injection: after planner completes, sprint steps are added
  7. Artifact validation: valid artifact passes, invalid artifact fails the step

**Dependencies:** None — this is the first step.

**Risks:** Mocking Prisma can be complex. Use `vitest.mock()` to mock `@prisma/client`. If the dispatcher has tight coupling to DB that makes mocking hard, note it but don't refactor the dispatcher now — just test what's testable.

**Definition of done:** At least 7 test cases pass covering the scenarios above. `npm run test` passes.

---

#### Step 0.2: Worker Tests

**Objective:** Test the worker's job execution lifecycle.

**Why it matters:** The worker picks jobs from the queue, runs them through runners, applies guardrails, tracks costs, and updates state. It's the second most critical file.

**What to change:** Create `src/bridge/worker.test.ts`.

**Suggested implementation approach:**
- Mock the runner factory (don't invoke real AI providers).
- Mock Prisma.
- Test these scenarios:
  1. Job pickup: worker selects the oldest `queued` job and marks it `running`
  2. Successful completion: runner returns output, job marked `completed`
  3. Runner failure: job marked `failed`, error recorded
  4. Budget gate: job blocked when budget exceeded, agent paused
  5. Guardrail pre-run: guardrail violation blocks execution
  6. Guardrail post-run: guardrail violation on output marks job failed
  7. Cost tracking: `CostEvent` created with token counts

**Dependencies:** None — can run in parallel with Step 0.1.

**Risks:** Worker may have side effects (file I/O, process spawning) that need isolation. Mock `fs` and `child_process` as needed.

**Definition of done:** At least 7 test cases pass. `npm run test` passes.

#### Step 1.1: Remove Bridge Routes

**Objective:** Delete `src/server/routes/bridge.ts` and all references to it.

**Why it matters:** Bridge routes are a parallel execution path that bypasses the pipeline system, guardrails, budget tracking, and artifact contracts. They allow direct agent invocation without any of the orchestration safety that the pipeline provides. Their continued existence means bugs can be "fixed" and features can be "built" without going through the system's own quality controls.

**What to change:**
1. Delete `src/server/routes/bridge.ts`.
2. In `src/server/routes/init.ts` (or wherever routes are registered), remove the import and registration of bridge routes.
3. Search for any references to `/api/review`, `/api/architect`, `/api/debug` in the codebase and remove them.
4. Search the MCP server (`src/mcp/index.ts`) for any tools that call bridge endpoints. If found, redirect them to use the intake/pipeline API instead, or remove the tool if it's redundant.

**Suggested implementation approach:**
```bash
# Find all references
grep -r "bridge" src/server/ --include="*.ts"
grep -r "/api/review\|/api/architect\|/api/debug" src/ --include="*.ts"
grep -r "buildReviewerPrompt\|buildArchitectPrompt\|buildDebuggerPrompt" src/ --include="*.ts"
```
Delete the file. Remove imports. Run `npm run lint` (tsc --noEmit) to find compilation errors. Fix each one.

**Dependencies:** Phase 0 complete (so we can run tests after).

**Risks:** If any MCP tool or CLI command depends on bridge routes, it will break. The search step above will reveal this. The fix is to redirect those calls to use `/api/intake` (the pipeline submission endpoint) instead.

**Definition of done:** `src/server/routes/bridge.ts` does not exist. `npm run lint` passes. `npm run test` passes. No code references bridge route paths.

### Phase 2: Context Flow (The Core Improvement)

This is the most important functional change in V2.3. It makes V1 fast pipelines actually useful.

---

#### Step 2.1: Wire Inter-Stage Context Flow in V1 Pipelines

**Objective:** Make each V1 pipeline stage receive the output of its predecessor stages as context input.

**Why it matters:** This is the single highest-impact change in V2.3. Currently, in fast mode (the default), the architect runs with the user's description, produces output, and then the builder runs with... only the user's description again. The architect's analysis, decisions, and recommendations are lost. This makes the multi-stage pipeline ceremonial. The stages exist but don't build on each other.

**What to change:**

The V2.1 harness already has `assembleHarnessStepContext()` in `src/orchestrator/harness-artifacts.ts` that assembles prior artifacts for harness steps. We need an equivalent for V1 classic steps.

1. Create a function `assembleClassicStepContext(pipelineRunId: string, currentStepKey: string): Promise<string>` in a suitable location (e.g., `src/orchestrator/context-assembler.ts` or add to the dispatcher).

2. This function should:
   - Query all `PipelineStepRun` records for the given `pipelineRunId` where `status === 'completed'` and the step's position in the pipeline is before `currentStepKey`.
   - For each completed prior step, extract its `resultSummary` (the human-readable output that agents produce).
   - Format them into a structured context block, e.g.:
     ```
     ## Prior Stage Outputs

     ### Architect (completed)
     <architect's resultSummary>

     ### Builder (completed)
     <builder's resultSummary>
     ```
   - Return this as a string.

3. In the dispatcher, when preparing a V1 (non-harness) step for execution, call `assembleClassicStepContext()` and append the result to the step's input context.

4. Specifically, find where the dispatcher creates the job for a pipeline step (likely in `enqueueEligibleSteps()` or the job creation logic). For non-harness runs, inject the assembled context into the job's input.

**Suggested implementation approach:**

Read `src/orchestrator/dispatcher.ts` and find:
- Where `PipelineStepRun` records are created or queued
- Where the job input/description is assembled for the runner
- The existing `assembleHarnessStepContext()` call — model the V1 version after this

The key insertion point is where the dispatcher transitions a step from "eligible" to "queued" and creates its corresponding job record. The job's description or input field should include the assembled context.

**Dependencies:** Phase 0 (tests exist so we can verify this works). Phase 1 complete (clean codebase to work in).

**Risks:**
- **Token bloat:** Prior stage outputs can be large. Add a per-stage output cap (e.g., 4000 chars) when assembling context. Truncate with a note: `[truncated to 4000 chars]`.
- **Circular context:** Ensure we only include stages that are *before* the current stage in the pipeline DAG, not stages that depend on the current stage.
- **Null outputs:** Some stages may complete with empty `resultSummary`. Skip them in the assembled context.

**Definition of done:**
- When a V1 feature pipeline runs, the builder's input includes the architect's output.
- When quality-guard runs, its input includes both architect and builder output.
- Verified by running a test pipeline and inspecting the job input in the database.
- Dispatcher tests updated to cover context assembly.

#### Step 2.2: Add Quality-Guard Feedback Loop to V1 Pipelines

**Objective:** Give the quality-guard stage the ability to reject and loop back to the builder.

**Why it matters:** Currently in V1 pipelines, the quality-guard runs, produces output, but has no enforcement mechanism. If it identifies issues, nothing happens — execution continues to devops. This makes the quality-guard pure theater. In V2.1 harness, feedback loops exist via `loopsBackTo` + `maxRevisions`. We need to bring this to V1.

**What to change:**

1. In the V1 pipeline definitions (`src/orchestrator/pipelines/`), add `loopsBackTo` and `maxRevisions` to the `quality-guard` step for feature, bug, and refactor pipelines.

   Example for the feature pipeline:
   ```typescript
   {
     key: 'quality-guard',
     agentSlug: 'reviewer',  // or whatever agent slug is used
     dependsOn: ['builder'],
     loopsBackTo: 'builder',
     maxRevisions: 2,
   }
   ```

2. Ensure the dispatcher's existing feedback loop logic works for V1 steps. The harness feedback loop logic in the dispatcher likely checks for `loopsBackTo` on the step definition. V1 steps should work the same way if we add this field.

3. The quality-guard agent's prompt (in `ai-system/official/agents/`) must be updated to produce a clear APPROVED/REJECTED signal. Check the current prompt. If it doesn't produce a structured verdict, add instructions like:
   ```
   End your review with one of:
   VERDICT: APPROVED — <one-line reason>
   VERDICT: REJECTED — <list of issues that must be fixed>
   ```

4. The dispatcher must parse this verdict to decide whether to loop back. Check if the existing feedback loop mechanism uses `resultSummary` parsing or a different signal. Align the quality-guard output format with whatever the dispatcher expects.

**Dependencies:** Step 2.1 (context flow must work first — the builder needs to receive the quality-guard's rejection feedback when it re-runs).

**Risks:**
- **Infinite loops:** `maxRevisions: 2` caps at 2 rejections. After that, the pipeline fails (hard stop). This is safe.
- **Verdict parsing:** If the dispatcher expects a specific format for loop-back decisions, the agent prompt must produce that format. Check the harness evaluator's verdict format and reuse it.
- **Agent behavior:** The quality-guard agent may not reliably produce APPROVED/REJECTED. This is a prompt quality issue, not an architecture issue. For V2.3, accept that the agent might not always format correctly — the `maxRevisions` cap protects against loops.

**Definition of done:**
- V1 feature/bug/refactor pipelines have `loopsBackTo: 'builder'` on quality-guard.
- When quality-guard rejects, the builder re-runs with the rejection feedback as context.
- After `maxRevisions` rejections, the pipeline fails (does not proceed).
- Dispatcher tests updated to cover V1 feedback loops.

## Concrete Architectural Decisions

These decisions are final for V2.3. Do not revisit them during implementation.

### Decision 1: One Pipeline System, Two Modes

V1 classic and V2.1 harness coexist. They share the same dispatcher. The `isStructuredPipelineRun()` check remains.

**Rationale:** Removing V1 entirely would force all users into multi-sprint structured mode, which is heavyweight for simple tasks. Both modes serve different use cases. V2.3 makes V1 mode actually useful (context flow + feedback loops) rather than removing it.

### Decision 2: Bridge Routes Are Deleted, Not Deprecated

Do not add deprecation warnings. Do not keep the file with a "deprecated" comment. Delete the file entirely.

**Rationale:** Deprecated code gets forgotten and re-discovered. The pipeline system is the only execution path. Bridge routes undermine that guarantee.

### Decision 3: Completions Endpoint Is Deleted

If someone needs OpenAI-compatible access in the future, it should be a separate adapter service, not embedded in the core server.

### Decision 4: TUI Is Primary, Web Is Secondary

The TUI console (`forge start`) is the primary developer interface. The web dashboard is a read-mostly monitoring surface. No new interactive features will be added to the web UI in V2.3.

### Decision 5: Single Default Company

The multi-company model is preserved in the database (needed for FK relationships), but the CLI and runtime auto-create and auto-select a default company. Users never need to think about company selection.

### Decision 6: Memory System Is Not Rewritten in V2.3

The `MemoryEntry` table and `context.md` system are acknowledged as weak, but rewriting them is out of scope. V2.3 focuses on intra-pipeline context flow (stage-to-stage), not cross-pipeline memory. Memory is a V2.4 concern.

### Decision 7: Context Assembly Has a Token Budget

When assembling prior stage outputs for V1 context flow (Step 2.1), each stage's output is capped at 4000 characters. Total assembled context is capped at 12000 characters. This prevents token bloat while providing meaningful context.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Removing bridge routes breaks MCP tools | High | Search MCP server for bridge endpoint references before deleting. Redirect to intake API if found. |
| Schema migration breaks existing databases | Medium | Run `npm run db:generate` after each schema change. Test with a fresh DB and an existing DB. |
| V1 context flow produces too much token input | Medium | Cap per-stage output at 4000 chars, total at 12000 chars. |
| Quality-guard feedback loop creates agent confusion | Medium | Cap at `maxRevisions: 2`. Accept that agent output may not always parse cleanly. |
| Removing `forge company` breaks MCP `companyId` resolution | Medium | Ensure MCP tools auto-resolve to default company. Test MCP tools after removal. |
| Dispatcher tests are hard to write due to DB coupling | Low | Use Prisma mocks. If too hard, write integration tests against SQLite in-memory. |
| Removing cloud sync leaves dead config references | Low | Search for "sync" references after deletion. Clean up. |

## Final Direction

V2.3 is not a glamorous release. It adds no new features. It does not make the product do more.

What it does is make the product **real**.

After V2.3, every pipeline stage in fast mode actually builds on the previous stage's output. Every UI surface the user sees is maintained and functional. Every execution path goes through the same orchestration system with the same guardrails, budget tracking, and artifact handling. Every table in the schema is used. Every command in the CLI does something meaningful and distinct.

The project's differentiator — typed artifact contracts, DAG-based pipeline orchestration, multi-provider runner abstraction, and interactive TUI — is strong. V2.3 protects that strength by removing the noise around it.

The next version (V2.4) can then focus on the real next frontier: cross-run memory intelligence, prompt testing, and making the structured harness mode the default. But V2.3 must come first, because you cannot build advanced features on a foundation that has three execution paths, zero tests on its critical path, and context flow that doesn't flow.

**Build the honest version first. Then build the ambitious version on top of it.**
